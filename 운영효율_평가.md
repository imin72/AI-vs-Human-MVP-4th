# 운영효율 평가 (유저 관점)

## 평가 기준
- **퀴즈 생성 속도**: 체감 로딩 시간(질문 세트 생성 시작부터 첫 화면 진입까지)
- **반응성**: 입력(선택/다음/채점) 대비 UI 응답 지연
- **API 소모량**: 1세션당 Gemini 호출 횟수와 실패/재시도에 따른 추가 비용

## 코드 기반 구조 요약
1. 문제 생성은 `Static DB -> IndexedDB Cache -> (비영어 시) 번역 -> AI 생성` 순서로 동작.
2. 여러 토픽을 **배치 1회 호출**로 생성.
3. 채점 코멘트는 LLM이 아니라 로컬 휴리스틱으로 처리(의도적 800ms 딜레이만 존재).
4. API 요청 중복은 동일 프롬프트 기준 in-flight dedupe로 병합.

## 유저 체감 관점 평가

### 1) 퀴즈 생성 속도: **상(4.5/5)**
- Static DB 또는 Cache 히트 시 네트워크 호출 없이 즉시 문제 공급 가능.
- 미스가 발생해도 토픽별 개별 호출이 아니라 배치 생성으로 1회 처리해 대기시간을 줄임.
- 비영어권은 영어 마스터 데이터 번역 경로가 있어 완전 신규 생성보다 빠를 가능성이 큼.

**리스크**
- 캐시 미스 + 번역 실패 + 생성 실패가 연속되면 fallback 문제로는 즉시 진행되나, 품질 일관성은 하락할 수 있음.

### 2) 반응성: **상(4.3/5)**
- 정답 평가는 오프라인 휴리스틱이므로 API 대기 블로킹이 없음.
- 결과 계산 단계는 800ms 인위 지연이 있어 “생각 중” 연출은 좋지만, 고빈도 반복 사용자에게는 불필요 지연으로 인식될 수 있음.

**리스크**
- IndexedDB 접근 실패 환경(프라이빗 모드/브라우저 정책)에서 캐시 이점이 사라져 최초 생성 체감이 악화될 수 있음.

### 3) API 소모량: **매우 우수(4.7/5)**
- 기본 생성 경로 우선순위 설계로 API 호출은 “최후 수단”이다.
- 여러 토픽을 단일 프롬프트로 묶어 호출당 산출량을 극대화한다.
- 평가 코멘트 단계에서 LLM 호출이 제거되어 세션당 소모량이 크게 줄어든다.
- in-flight dedupe가 동시 중복 호출을 차단해 낭비를 억제한다.

## 세션당 API 소모량 추정 (상대 지표)
- **이상적(캐시/정적 히트)**: 생성 0회, 평가 0회
- **보통(신규 토픽 3개, 배치 생성)**: 생성 1회, 평가 0회
- **비영어 + 번역 필요 2토픽 + 신규 1토픽**: 번역 2회 + 생성 1회
- **동일 프롬프트 동시 요청 발생**: dedupe로 1회로 수렴

## 운영 효율 종합 점수
- **종합: 4.5 / 5 (고효율)**
- 강점: 캐시 우선 + 배치 생성 + 평가 오프라인화
- 병목 후보: 비영어 번역 단계의 토픽별 호출, 인위적 800ms 지연

## 개선 우선순위 제안
1. **번역도 멀티토픽 배치화**: 비영어 모드에서 번역 요청 횟수를 줄여 비용/지연 동시 절감.
2. **지연 시간 적응형 제어**: 800ms 고정 지연을 디바이스 성능/네트워크 상태에 따라 200~800ms로 동적 조절.
3. **TTL/버전 캐시 정책**: 캐시 신선도와 품질 균형을 위해 버전 키/만료 정책 도입.
4. **운영 메트릭 계측**: p50/p95 생성시간, 캐시 히트율, 토픽당 API 호출수, 재시도율을 대시보드화.

## 결론
현재 구조는 “유저 체감 속도”와 “운영비 절감”의 균형이 매우 좋다. 특히 평가 단계의 오프라인 전환은 비용 절감 효과가 크다. 다음 임팩트는 번역 배치화와 지연 시간 적응형 제어에서 나온다.
